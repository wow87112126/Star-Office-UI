# Star Office UI 实时状态桥接方案（Step 9）

## 目标
让公网 `https://office.ziwofuxian.com` 不只是一个静态上线页面，而是能持续反映**本地 OpenClaw / Hermes 的真实工作状态**，并支持主人离开办公环境后，继续通过飞书群聊给 bot 下达指令，从而驱动本地工作与宠物状态联动。

---

## 一、三层架构（推荐长期形态）

### 1. 控制层（Command Plane）
**飞书群聊 → Hermes / OpenClaw**

作用：
- 主人远程下达任务
- bot 接收指令
- Hermes / OpenClaw 继续执行工作

当前基础：
- 飞书 bot 已接入 Hermes Gateway
- 关键词唤醒 / 群内交互已打通

### 2. 执行层（Execution Plane）
**本地 OpenClaw / Hermes 主会话**

作用：
- 真正处理任务
- 写本地状态
- 维护项目上下文
- 执行修改、调研、同步、回复等工作

### 3. 展示层（Display Plane）
**公网 Star Office UI 服务器**

作用：
- 向外展示宠物动作 / 当前状态 / 办公室看板
- 不直接承载本地 OpenClaw 本体
- 只接受状态同步结果

---

## 二、为什么当前还不能自动同步
当前线上站点和本地 OpenClaw 是两个独立环境：

- 本地：`C:\Users\Administrator\.openclaw\workspace\Star-Office-UI`
- 服务器：`/home/ecs-user/Star-Office-UI`

当前已经完成的是：
- 代码部署到服务器
- Caddy + Gunicorn + systemd 常驻运行

但尚未完成的是：
- 本地状态桥接到公网服务器

所以现在：
- 本地任务状态变化不会自动传到线上
- 飞书里下达的新任务，也不会自动推动公网宠物动作变化

---

## 三、推荐的一步到位方向

### 核心原则
**本地 OpenClaw / Hermes 继续做“大脑”，公网 Star Office UI 只做“展示面板”。**

不要把整个本地 OpenClaw 暴露到公网；而是把“状态”安全地推送到公网。

### 推荐链路
飞书群聊主人指令
→ Hermes Gateway 接收
→ 本地 OpenClaw / Hermes 开始工作
→ 写入本地 `state.json`
→ 本地状态桥接器读取状态
→ 推送到公网 `office.ziwofuxian.com`
→ 公网宠物动作实时变化

---

## 四、最适合当前项目的实现方式
项目本身已经内置：
- `office-agent-push.py`
- `/join-agent`
- `/agent-push`
- `/leave-agent`

这意味着最自然的方案不是自己重造协议，而是：

### 方案：把“本地 OpenClaw / Hermes”视为一个 remote agent
本地状态桥接器负责：
1. 读取本地 `state.json`
2. 自动加入公网办公室（`join-agent`）
3. 周期推送状态（`agent-push`）
4. 当状态变化时更新宠物动作 / 区域 / 气泡

这就是后续最优先要落地的功能。

---

## 五、和飞书指令的联动关系
主人以后离开办公室后，通过飞书群聊发指令，理想链路应是：

飞书群聊
→ Hermes bot 收到任务
→ Hermes / OpenClaw 在本机继续工作
→ 在任务开始、执行中、同步中、报错、完成时更新本地状态
→ 状态桥接器把这些状态推送到公网办公室
→ 主人远程打开 `office.ziwofuxian.com` 就能看到宠物动作同步变化

也就是说：
- **飞书负责下命令**
- **Hermes 负责执行**
- **Star Office UI 负责展示状态**

三者分工明确。

---

## 六、必须明确的现实条件
要实现“主人出门后，飞书继续指挥本地工作”，至少要满足：

### 条件 1：本地承载 OpenClaw / Hermes 的机器必须在线
如果机器：
- 睡眠
- 关机
- 断网

那么飞书指令虽然能发出来，但本地不会继续执行。

### 条件 2：Hermes / OpenClaw 网关必须持续运行
否则飞书 bot 收不到后续任务。

### 条件 3：状态桥接器必须持续运行
否则就算任务在做，公网宠物状态也不会同步变化。
